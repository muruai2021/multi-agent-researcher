# Multi-Agent Research 流水线详解（主编调度手册）

> 本文档是 [SKILL.md](../SKILL.md) 的完整流水线细节，供主编调度时参考。

---

## 四阶段流水线概览

```
用户说"帮我调研XX市场"
         ↓
主编命题解析 + Scout配置
         ↓
━━━━━━━━━━━━━━━━━━━━━━━━
阶段一：并行侦察（5个Scout）
Scout-A → 市场数据包
Scout-B → 竞品数据包
Scout-C → 用户数据包
Scout-D → 客户数据包
Scout-E → 案例数据包
━━━━━━━━━━━━━━━━━━━━━━━━
         ↓ Checkpoint CP-1
阶段二：分析整合
Analyst → 市场洞察报告
Critic  → 风险与机会矩阵
         ↓ Checkpoint CP-2
阶段三：报告撰写
Writer  → 完整报告初稿
         ↓ Checkpoint CP-3
阶段四：评审定稿
Reviewer-A → 数据/逻辑评审
Reviewer-B → 可行性评审
         ↓ Checkpoint CP-4
   ┌────┴────┐
   ↓         ↓
 ✅ 通过   ❌ 不通过
 定稿    Writer修订
              ↑
       （循环直到达标或3轮上限）
```

## 主编调度详解（含调度恢复）

### 阶段零：命题解析（CP-0 触发）

**主编任务：**
1. 解析用户调研请求，明确调研命题
2. 判断调研类型（通用商业/产品/竞品/用户研究）
3. 判断调研深度（快速扫描/标准/深度）
4. 选择 Scout 配置组合
5. **触发 CP-0**：持久化命题、配置、深度

**决策树：**
```
用户请求
  ↓
是否模糊？
  ├── 是 → 追问用户（最多3次）
  └── 否 ↓
是否在 Not For 列表中？
  ├── 是 → 明确说明限制 + 转介
  └── 否 ↓
选择调研类型与深度
  ↓
CP-0 触发 ✓
```

### 阶段一：并行侦察

**主编任务：**
1. 准备项目背景 briefing 文档
2. 选择 Scout 配置（按调研类型）
3. 并行下发调研任务给 5 个 Scout
4. 等待 5 个数据包全部返回（**调度超时 2 分钟 / 工作超时 30 分钟**）
5. 验证数据完整性
6. **触发 CP-1**：持久化 5 个数据包 + 缺失标记

**并行执行规范：**
- 5 个 Scout **严格并行**，不串行
- 主编同步等待全部返回
- 任何一个超时即触发 L1 失败处理（见下文"失败模式与降级矩阵"）
- 不允许"先等 3 个就开始下一阶段"——必须全到或超时降级

**Scout配置选择：**

| 调研类型 | Scout-A | Scout-B | Scout-C | Scout-D | Scout-E |
|----------|---------|---------|---------|---------|---------|
| 通用商业 | ✅市场 | ✅竞品 | ✅用户 | ✅客户 | ✅案例 |
| 产品调研 | ✅需求 | ✅竞品 | ✅用户 | ✅反馈 | ✅产品案例 |
| 竞品研究 | ✅份额 | ✅产品 | ✅评价 | ✅动机 | ✅失败案例 |
| 用户研究 | ✅画像 | ✅行为 | ✅需求 | ✅流失 | ✅高满意 |

### 阶段二：分析整合

**主编任务：**
1. 将 5 个数据包整理成统一格式
2. 下发 Analyst 整合任务
3. 下发 Critic 质疑任务
4. 等待洞察报告+风险矩阵（**并行**）
5. 评估整合质量
6. **触发 CP-2**：持久化 Analyst + Critic 输出

**并行执行：**
```
Analyst → 洞察整合（数据去重+洞察提炼+机会矩阵）
Critic  → 风险质疑（假设挑战+风险矩阵+时机评估）
```

**降级路径（关键）：**
- Analyst 失败 → 主编用 Scout 原始数据直出洞察初稿
- Critic 失败 → 主编代行质疑（基于经验 + 触发"挑战假设"）
- 两者都失败 → 触发 Checkpoint + 询问用户

### 阶段三：报告撰写

**主编任务：**
1. 整理所有素材（Scout + Analyst + Critic）
2. 下发 Writer 撰写任务
3. 等待报告初稿（**工作超时 2-3h**）
4. 审核报告结构完整性
5. **触发 CP-3**：持久化报告初稿

**降级路径：**
- Writer 失败 → 主编用 Scout + Analyst 输出 + 模板拼接 Markdown 简版
- 标记"非完整版"告知用户

### 阶段四：评审定稿

**主编任务：**
1. 下发 Reviewer-A 数据审计任务
2. 下发 Reviewer-B 可行性评估任务
3. 汇总评审意见
4. 判断是否需要修订
5. **触发 CP-4**：持久化评审结果 + 定稿状态

**评审决策：**
```
评审意见汇总
     ↓
  通过？
    ↓
 ┌──┴──┐
 ↓     ↓
✅    ❌
定稿  Writer修订
       ↑
  （最多3轮循环）
  （超出则交付当前版本 + 剩余问题清单）
```

**降级路径：**
- Reviewer-A 失败 → 跳过数据审计轮次，标记"数据未审计"
- Reviewer-B 失败 → 跳过可行性轮次，标记"可行性未评估"
- 两者都失败 → 标记"评审未完成"，询问用户是否接受无评审交付

## Checkpoint 机制详解

### Checkpoint 触发矩阵

| 编号 | 触发时机 | 持久化内容 | 下一阶段入口 |
|------|----------|-----------|--------------|
| CP-0 | 命题解析完成 | 命题、类型、深度、配置 | 阶段一 |
| CP-1 | 5 个 Scout 数据包齐 | 数据包 + 缺失标记 | 阶段二 |
| CP-2 | Analyst + Critic 完成 | 洞察 + 风险矩阵 | 阶段三 |
| CP-3 | Writer 报告初稿完成 | 报告初稿 | 阶段四 |
| CP-4 | 评审通过 / 3 轮上限 | 评审结果 + 定稿状态 | 交付 |

### 存储规范

```
references/sessions/{project_id}/
├── checkpoint-0.json    # CP-0
├── checkpoint-1.json    # CP-1
├── checkpoint-2.json    # CP-2
├── checkpoint-3.json    # CP-3
└── checkpoint-4.json    # CP-4
```

**每个 Checkpoint JSON 格式：**
```json
{
  "checkpoint_id": "CP-1",
  "project_id": "ai-coding-cn-2026q2",
  "triggered_at": "2026-06-15T14:32:00+08:00",
  "stage_name": "阶段一：并行侦察",
  "outputs": {
    "scout_a_market": { "hash": "sha256:...", "size": 4321 },
    "scout_b_competitor": { "hash": "sha256:...", "size": 3892 },
    ...
  },
  "next_stage": "阶段二：分析整合",
  "next_agents": ["Analyst", "Critic"]
}
```

### 恢复流程

```
工作中断（断电/超时/Token 耗尽/用户取消）
  ↓
主编重新激活
  ↓
扫描 references/sessions/ 找到最近 Checkpoint
  ↓
读取 Checkpoint JSON，验证 outputs 哈希
  ↓
确认 next_stage，从该阶段重入
  ↓
复用已完成产出，仅执行 next_agents
```

### Checkpoint 清理

- 项目交付后 30 天自动归档
- 归档路径 `references/sessions/_archive/{project_id}/`
- 6 个月后可删除

## 失败模式与降级矩阵（4 类 L1-L4）

| 失败类型 | 检测信号 | 默认重试 | 降级路径 | 决策人 | 报告标记 |
|----------|----------|----------|----------|--------|----------|
| **L1 调度失败** | 调度超时 2 分钟、心跳丢失 | 1 次 | 重试失败则跳到 L2 | 主编 | `[L1: 跳过]` |
| **L2 数据缺失** | Scout 数据包为空/不完整 | 0 次 | 标记缺口 + 用其他 Scout 旁证 | 主编 | `[L2: 数据缺口]` |
| **L3 输出不合格** | Reviewer 标 Critical / High > 2 | 0 次 | Writer 修订（最多 3 轮） | 主编 | `[L3: 修订 N 轮]` |
| **L4 资源耗尽** | Token 超限、API 速率限制 | 0 次 | 保存 CP + 暂停 + 询问用户 | 主编 + 用户 | `[L4: 暂停]` |

### 非 Scout Agent 失败降级

| Agent | 默认影响 | 降级策略 | 报告标记 |
|-------|----------|----------|----------|
| **Analyst** | 无洞察整合 | 主编用 Scout 数据直出洞察初稿 | `[降级: 主编代行洞察]` |
| **Critic** | 无风险质疑 | 主编代行质疑（基于经验） | `[降级: 主编代行质疑]` |
| **Writer** | 无报告 | 主编用模板拼接 Markdown 简版 | `[降级: 简版报告]` |
| **Reviewer-A** | 无数据审计 | 跳过数据审计 | `[降级: 数据未审计]` |
| **Reviewer-B** | 无可行性评估 | 跳过可行性评估 | `[降级: 可行性未评估]` |
| **2+ Agent 同时失败** | 质量不可控 | 保存 CP + 暂停 + 询问用户 | `[L4: 暂停]` |

## 超时分层规范（修正 LOG/STA 歧义）

| 层级 | 类型 | 默认阈值 | 含义 | 超时动作 |
|------|------|----------|------|----------|
| T1 | 调度超时 | 2 分钟 | Agent 启动 / 心跳超时 | 重试 1 次 → 标记 L1 |
| T2 | 工作超时 | 30 分钟 | 单个 Scout 单次任务执行 | 跳过 + 标记数据缺口 |
| T3 | 阶段超时 | 视阶段 | Scout 2-3h / Analyst 1-2h / Writer 2-3h / Reviewer 1-2h | 触发 Checkpoint + 询问用户 |
| T4 | 总超时 | 24 小时 | 整个调研流程 | 强制交付当前版本 + 剩余问题清单 |

**T3 阶段超时明细：**

| 阶段 | 标准时长 | 深度模式 | 快速模式 | 超时动作 |
|------|----------|----------|----------|----------|
| 阶段一（5 Scout 并行） | 2-3h | 4h | 1h | CP-1 + 询问用户 |
| 阶段二（Analyst + Critic 并行） | 1-2h | 3h | 30min | CP-2 + 询问用户 |
| 阶段三（Writer 串行） | 2-3h | 4h | 1h | CP-3 + 询问用户 |
| 阶段四（2 Reviewer 并行） | 1-2h | 3h | 30min | CP-4 + 询问用户 |

## 数据质量评估

主编在 Scout 数据包返回后进行质量检查：

| 检查项 | 标准 | 不合格处理 |
|--------|------|-----------|
| 数据来源 | 有明确来源标注 | 要求 Scout 补充 |
| 数据时效 | 3 年内数据 | 标注时效性 |
| 数据交叉验证 | 多源一致 | 标注矛盾点 |
| 数据完整性 | 核心指标齐全 | 要求补充缺口 |

---

## 评审通过标准

| 评审项 | Reviewer-A（数据审计） | Reviewer-B（落地评估） |
|--------|----------------------|----------------------|
| Critical 问题数 | 0 | 0 |
| High 问题数 | ≤2 | ≤2 |
| 总体判定 | 通过/需修订 | 通过/需修订 |

**Combined Decision:**
- 两个 Reviewer 都通过 → ✅ 定稿
- 任一 Reviewer 有 Critical → ❌ 修订
- 两个 Reviewer 都有 High → ❌ 修订
- 修订达 3 轮 → ⚠️ 交付当前版本，说明剩余问题

## 时间估算

| 阶段 | Scout(N) | Analyst | Critic | Writer | Reviewer(N) |
|------|----------|---------|--------|--------|-------------|
| 阶段一 | 2-3h（并行） | - | - | - | - |
| 阶段二 | - | 1-2h | 1h | - | - |
| 阶段三 | - | - | - | 2-3h | - |
| 阶段四 | - | - | - | 1-2h | 1-2h |

**总计：6-10小时**

| 复杂度 | 时间 | 适用场景 |
|--------|------|----------|
| 简单 | 4-6h | 单一市场/快速扫描 |
| 标准 | 6-10h | 多维度综合调研 |
| 复杂 | 12-20h | 跨国/多产品/深度 |

## 主编检查清单（升级版）

### 阶段零完成后检查
- [ ] CP-0 已触发，命题解析完整
- [ ] 调研类型与深度明确
- [ ] Scout 配置已选定
- [ ] 用户已确认（如有追问）

### 阶段一完成后检查
- [ ] Scout-A 数据包：市场规模 + CAGR + 趋势
- [ ] Scout-B 数据包：竞品列表 + 份额 + 优劣势
- [ ] Scout-C 数据包：用户画像 + 需求 + 行为
- [ ] Scout-D 数据包：客户结构 + 满意度（如适用）
- [ ] Scout-E 数据包：成功 + 失败案例各≥1
- [ ] **CP-1 已触发，5 个数据包已持久化**
- [ ] 缺失数据包已标记 L2

### 阶段二完成后检查
- [ ] Analyst 洞察报告：5 个洞察以内
- [ ] Analyst 机会矩阵：推荐度明确
- [ ] Critic 风险矩阵：P0 风险有应对
- [ ] Critic 整体判断：可行/有条件/不可行
- [ ] **CP-2 已触发**
- [ ] Analyst/Critic 降级情况已记录（如有）

### 阶段三完成后检查
- [ ] 执行摘要：核心结论明确
- [ ] 8 章结构：完整无缺章
- [ ] 每章小结：有行动建议
- [ ] 数据支撑：每结论有数据
- [ ] **CP-3 已触发**
- [ ] Writer 降级情况已记录（如有）

### 阶段四完成后检查
- [ ] Critical 问题：全部解决
- [ ] High 问题：≤2 个
- [ ] 报告定稿：可对外发布
- [ ] **CP-4 已触发**
- [ ] Reviewer 降级情况已记录（如有）

### 整体交付前检查
- [ ] 报告路径：references/output/{project_id}/report.html
- [ ] 失败案例已更新到 failure_case_log
- [ ] 用户已收到交付通知
- [ ] 自我迭代动作已触发（参见 SKILL.md）
