# Multi-Agent Research 流水线详解（主编调度手册）

> 本文档是 [SKILL.md](SKILL.md) 的完整流水线细节，供主编调度时参考。

---

## 四阶段流水线概览

```
用户说"帮我调研XX市场"
         ↓
主编命题解析 + Scout配置
         ↓
━━━━━━━━━━━━━━━━━━━━━━━━
阶段一：并行侦察（5个Scout）
Scout-A → 市场数据包
Scout-B → 竞品数据包
Scout-C → 用户数据包
Scout-D → 客户数据包
Scout-E → 案例数据包
━━━━━━━━━━━━━━━━━━━━━━━━
         ↓
阶段二：分析整合
Analyst → 市场洞察报告
Critic  → 风险与机会矩阵
         ↓
阶段三：报告撰写
Writer  → 完整报告初稿
         ↓
阶段四：评审定稿
Reviewer-A → 数据/逻辑评审
Reviewer-B → 可行性评审
         ↓
   ┌────┴────┐
   ↓         ↓
 ✅ 通过   ❌ 不通过
 定稿    Writer修订
              ↑
       （循环直到达标或3轮上限）
```

## 主编调度详解

### 阶段一：并行侦察

**主编任务：**
1. 准备项目背景briefing文档
2. 选择Scout配置（按调研类型）
3. 并行下发调研任务给5个Scout
4. 等待5个数据包全部返回
5. 验证数据完整性

**Scout配置选择：**

| 调研类型 | Scout-A | Scout-B | Scout-C | Scout-D | Scout-E |
|----------|---------|---------|---------|---------|---------|
| 通用商业 | ✅市场 | ✅竞品 | ✅用户 | ✅客户 | ✅案例 |
| 产品调研 | ✅需求 | ✅竞品 | ✅用户 | ✅反馈 | ✅产品案例 |
| 竞品研究 | ✅份额 | ✅产品 | ✅评价 | ✅动机 | ✅失败案例 |
| 用户研究 | ✅画像 | ✅行为 | ✅需求 | ✅流失 | ✅高满意 |

### 阶段二：分析整合

**主编任务：**
1. 将5个数据包整理成统一格式
2. 下发Analyst整合任务
3. 下发Critic质疑任务
4. 等待洞察报告+风险矩阵
5. 评估整合质量

**并行执行：**
```
Analyst → 洞察整合（数据去重+洞察提炼+机会矩阵）
Critic  → 风险质疑（假设挑战+风险矩阵+时机评估）
```

### 阶段三：报告撰写

**主编任务：**
1. 整理所有素材（Scout数据包+洞察报告+风险矩阵）
2. 下发Writer撰写任务
3. 等待报告初稿
4. 审核报告结构完整性

### 阶段四：评审定稿

**主编任务：**
1. 下发Reviewer-A数据审计任务
2. 下发Reviewer-B可行性评估任务
3. 汇总评审意见
4. 判断是否需要修订

**评审决策：**
```
评审意见汇总
     ↓
  通过？
    ↓
 ┌──┴──┐
 ↓     ↓
✅    ❌
定稿  Writer修订
       ↑
  （最多3轮循环）
```

## Scout数据包格式标准

主编统一整理后发给Analyst：

```yaml
# Scout数据包汇总
project: [项目名称]
research_date: [日期]

scout_a_market:
  数据点: [...]
  规律: [...]
  机会: [...]
  风险: [...]
  数据缺口: [...]

scout_b_competitor:
  直接竞品: [...]
  间接竞品: [...]
  竞争格局: [...]
  机会窗口: [...]
  威胁边界: [...]

scout_c_user:
  用户画像: [...]
  行为特征: [...]
  核心需求: [...]
  未被满足: [...]
  支付意愿: [...]

scout_d_customer:
  客户结构: [...]
  消费行为: [...]
  满意度: [...]
  核心期望: [...]

scout_e_case:
  成功案例: [...]
  失败案例: [...]
  案例启示: [...]
```

## 数据质量评估

主编在Scout数据包返回后进行质量检查：

| 检查项 | 标准 | 不合格处理 |
|--------|------|-----------|
| 数据来源 | 有明确来源标注 | 要求Scout补充 |
| 数据时效 | 3年内数据 | 标注时效性 |
| 数据交叉验证 | 多源一致 | 标注矛盾点 |
| 数据完整性 | 核心指标齐全 | 要求补充缺口 |

---

## 评审通过标准

| 评审项 | Reviewer-A（数据审计） | Reviewer-B（落地评估） |
|--------|----------------------|----------------------|
| Critical问题数 | 0 | 0 |
| High问题数 | ≤2 | ≤2 |
| 总体判定 | 通过/需修订 | 通过/需修订 |

**Combined Decision:**
- 两个Reviewer都通过 → ✅ 定稿
- 任一Reviewer有Critical → ❌ 修订
- 两个Reviewer都有High → ❌ 修订
- 修订达3轮 → ⚠️ 交付当前版本，说明剩余问题

## 时间估算

| 阶段 | Scout(N) | Analyst | Critic | Writer | Reviewer(N) |
|------|----------|---------|--------|--------|-------------|
| 阶段一 | 2-3h（并行） | - | - | - | - |
| 阶段二 | - | 1-2h | 1h | - | - |
| 阶段三 | - | - | - | 2-3h | - |
| 阶段四 | - | - | - | 1-2h | 1-2h |

**总计：6-10小时**

| 复杂度 | 时间 | 适用场景 |
|--------|------|---------|
| 简单 | 4-6h | 单一市场/快速扫描 |
| 标准 | 6-10h | 多维度综合调研 |
| 复杂 | 12-20h | 跨国/多产品/深度 |

## 主编检查清单

### 阶段一完成后检查
- [ ] Scout-A数据包：市场规模+CAGR+趋势
- [ ] Scout-B数据包：竞品列表+份额+优劣势
- [ ] Scout-C数据包：用户画像+需求+行为
- [ ] Scout-D数据包：客户结构+满意度（如适用）
- [ ] Scout-E数据包：成功+失败案例各≥1

### 阶段二完成后检查
- [ ] Analyst洞察报告：5个洞察以内
- [ ] Analyst机会矩阵：推荐度明确
- [ ] Critic风险矩阵：P0风险有应对
- [ ] Critic整体判断：可行/有条件/不可行

### 阶段三完成后检查
- [ ] 执行摘要：核心结论明确
- [ ] 8章结构：完整无缺章
- [ ] 每章小结：有行动建议
- [ ] 数据支撑：每结论有数据

### 阶段四完成后检查
- [ ] Critical问题：全部解决
- [ ] High问题：≤2个
- [ ] 报告定稿：可对外发布
